# Octra Identity

Octra Identity is an on-chain naming protocol and consumer dApp for human-readable `.oct` identities on the Octra network. It combines an AML smart contract with a responsive React client that can run as a conventional Web3 dApp or inside an Octra Circle.

## Features

- Search and register `.oct` names.
- Resolve root names and subdomains to Octra or Circle addresses.
- Set a primary name for a wallet.
- Update destinations without transferring ownership.
- Create, update, and release subdomains.
- Transfer or release owned names.
- List names in the marketplace and manage listings.
- Discover and connect RFC-O-1-compatible wallets.
- Build separate Web3 and Circle frontend variants.
- Compile, statically verify, deploy, publish ABI metadata, and verify AML source.

## Read Performance

Octra Identity keeps enumerable owner, marketplace, and subdomain indexes in
contract state. Read-heavy clients consume those indexes through versioned
snapshot views:

- `get_name_snapshot` returns every field required for an exact-name result in
  one view.
- `get_owner_page`, `get_listing_page`, and `get_subdomain_page` return up to
  25 records per call.
- `get_config_snapshot` replaces the individual configuration view fan-out.
- Per-owner, listing, registry, config, and subdomain versions let clients
  validate a local cache without downloading unchanged records again.

Snapshot page zero establishes the version and total. Later pages provide that
version as a guard; the contract returns a stale marker if a concurrent
swap-and-pop mutation invalidates the cursor. Clients then restart from page
zero. This keeps reads bounded and prevents duplicate or skipped records.
`VITE_ONS_PAGE_SIZE` may lower the client page size from the default 25 when a
constrained gateway benefits from smaller responses; the contract always
enforces the 25-row maximum.

The frontend renders a cached My Names snapshot immediately, validates it
against the owner version, and only downloads pages after a relevant on-chain
change. Frontend builds retain legacy view fallbacks so they can remain usable
during a staged contract rollout.

## Repository Structure

```text
ons.aml             Octra Identity AML contract
scripts/            Contract compile, deploy, verification, and test tools
frontend/           React and Vite consumer dApp
```

Generated build artifacts, deployment records, local environment files, dependencies, logs, and wallet material are intentionally excluded from version control.

## Requirements

- Node.js 18 or newer.
- Access to a compatible Octra JSON-RPC endpoint.
- An RFC-O-1-compatible wallet for dApp transactions.
- A dedicated deployer wallet with sufficient OCT for contract deployment when publishing a contract.

## Contract Development

Install the contract tooling dependencies:

```bash
npm install
```

Create a local `.env` from `.env.example`, then configure the target network and RPC. `AML_DEPLOYER_ADDR` and `AML_DEPLOYER_PRIV` are required only for signed deployment or test operations. Keep those values local and use a dedicated deployer wallet.

Compile and run static verification:

```bash
npm run compile
npm run static-verify
```

Deploying submits an on-chain transaction. Review the selected RPC, network, predicted contract address, constructor arguments, fee, and deployer before continuing:

```bash
npm run deploy
npm run save-abi
npm run verify
npm test
```

The toolchain writes generated bytecode, ABI, deployment metadata, and verification output under `build/`. These files are local artifacts and are not committed.

## Frontend Development

```bash
cd frontend
npm ci
npm run dev:web3:devnet
```

Available development targets:

```bash
npm run dev:web3:devnet
npm run dev:web3:mainnet
npm run dev:circle:devnet
npm run dev:circle:mainnet
```

Frontend configuration is documented in `frontend/.env.example`. Every `VITE_*` value is public because Vite embeds it in the browser bundle. Never place private keys, API secrets, or privileged credentials in frontend environment variables.

## Production Builds

Build the Web3 client:

```bash
cd frontend
npm ci
npm run build:web3:mainnet
```

Build the Circle client:

```bash
cd frontend
npm ci
npm run build:circle:mainnet
```

Both commands produce static files in `frontend/dist/`.

### Web Deployment

Serve `frontend/dist/` from an HTTPS static host. Configure the selected contract address, network, RPC, explorer, and base path before building. The host should return `index.html` for client-side routes and apply an appropriate Content Security Policy for the configured RPC and wallet integrations.

### Circle Deployment

Upload the Circle build output while preserving paths relative to `frontend/dist/`. Use a Circle profile and resource mode appropriate for the application, then wait for each asset transaction to be confirmed before publishing the final `oct://` entry point. Circle deployment credentials and deployment metadata should remain outside this repository.

## Security

- Never commit `.env` files, deployer keys, seed phrases, wallet exports, deployment receipts, or logs.
- Use a separate, minimally funded deployer wallet.
- Verify the active network and contract address before every signed operation.
- Treat frontend configuration as public information.
- Run compile and static verification before deployment, then verify the deployed source and ABI.
- Review contract and frontend changes independently before publishing a release.

## Contract Validation

The contract validates lowercase labels using `a-z`, `0-9`, and an internal hyphen. A label cannot begin or end with a hyphen. Subdomain labels follow the same shape and may be one character long.
